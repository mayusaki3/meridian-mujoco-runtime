# meridian-mujoco-runtime ロードマップ案：KONDO_KRS-2552 sysid から最終形まで

## 1. 前提

## 1.1 対象システム

仮称：

```text
meridian-mujoco-runtime
```

目的：

```text
MuJoCo上で、実機ロボット・アクチュエーター・フィールド・制御ボードを再現し、
上位アプリや実ボードと連携できるシミュレーション実行環境を構築する。
```

---

## 1.2 今回のマイルストーン1対象

```text
KONDO_KRS-2552 系サーボ単体
```

主目的：

```text
負荷時・無負荷時のKONDO_KRS-2552サーボ単体挙動を、
MuJoCo上でほぼ正しく再現できるようにする。
```

前提：

```text
サーボ設定値は、KHR-3HVで使われる実用的な推奨値を基準とする。
最大値の80%程度を安全側の固定値として使ってよい。
```

---

# 2. 現状要件への当てはめ評価

## 2.1 小目標への適合

小目標：

```text
負荷時、無負荷時のKONDO_KRS-2552サーボ単体がMuJoCoで再現したとき、
挙動がほぼ正しく再現されるようにする。
```

これは、これまで検討した以下の構成で対応可能。

```text
指令ログ
外部角度センサー
外部電流・電圧センサー
簡易負荷治具
sysid処理
MuJoCo ActuatorBehaviorModel
```

必要なモデル：

```text
位置制御モデル
速度制限
トルク制限
保持剛性
摩擦
デッドバンド
応答遅れ
初動補正
```

マイルストーン1として妥当。

---

## 2.2 中目標への適合

中目標：

```text
実機の歩行モーションが、シミュレーター上でも同じように再現される。
シミュレータ上で歩けているモーションが、実機上でも同じように再現される。
```

これは、サーボ単体sysidだけでは不足する。

追加で必要：

```text
KHR-3HV全体のロボットモデル
各関節へのサーボ割り当て
リンク質量・重心・慣性
足裏接触モデル
床摩擦
関節可動域
配線・電源電圧降下
歩行モーション再生機能
実機ログ収集
シミュレーションログ比較
```

特に歩行では、以下が重要。

```text
足裏接触
床摩擦
重心位置
リンク慣性
サーボ間の電源共有
制御周期
モーション補間
```

したがって、中目標はマイルストーン2以降に分離するのが妥当。

---

## 2.3 大目標への適合

大目標：

```text
パラメータを変化させた時も、同様の結果となるパラメータの関数式によるテンプレートが完成している。
誰もが試せるパラメータ同定を自動化する仕組みが完成し、他のサーボのパラメータも自動で特定できる。
```

これは、単一サーボの固定設定sysidより上位の目標である。

追加で必要：

```text
サーボ設定値を説明変数とした回帰モデル
複数設定値での測定データ
実験計画自動生成
測定手順自動化
データ品質判定
sysid自動実行
モデル選択
検証レポート自動生成
他サーボ用capability profile
```

このため、大目標はマイルストーン4以降に置くべき。

# 4. 追加すべき要件詳細

## 4.1 再現精度の判定基準

「ほぼ正しく」を定量化する必要がある。

例：

```text
無負荷ステップ応答:
  角度RMSE <= 目標値
  整定時間誤差 <= 目標値
  最大速度誤差 <= 目標値

負荷保持:
  角度沈み込み誤差 <= 目標値
  保持限界トルク誤差 <= 目標値

歩行:
  足先軌道誤差 <= 目標値
  重心軌道誤差 <= 目標値
  転倒有無一致
```

初期目標値は仮でよい。

例：

```text
角度RMSE: 5度以内
整定時間誤差: 20%以内
最大速度誤差: 20%以内
保持トルク誤差: 25%以内
```

マイルストーン1では、厳密一致ではなく「実用的な近似」を目標にする。

---

## 4.2 推奨サーボ設定値の正本管理

課題として挙げられている通り、サーボ設定値を変えると挙動が複雑に変化する。

そのため、最初は以下を固定する。

```text
speed
stretch
punch
deadband
response
damping
current limit
temperature limit
リミット
通信速度
制御周期
```

要件：

```text
KRS-2552推奨設定プロファイルを1つ作る
測定データには必ず設定プロファイルIDを付与する
sysid結果にも設定プロファイルIDを紐づける
```

---

## 4.3 測定条件の標準化

同じサーボでも条件が変わると結果が変わる。

固定すべき条件：

```text
電源電圧
電流制限
周囲温度
サーボ温度
負荷アーム長
負荷質量
取付姿勢
初期角度
指令周期
測定周期
サーボ設定値
```

---

## 4.4 負荷治具の標準仕様

マイルストーン1では、複雑な治具より単純なものがよい。

推奨：

```text
既知長アーム + 重り
```

追加で可能なら：

```text
ロードセル
```

最低限必要な定義：

```text
arm_length_m
mass_kg
gravity_m_s2
load_angle_rad
estimated_torque_nm
```

---

## 4.5 外部センサー要件

サーボ内部値だけでは不十分。

最低限：

```text
外部角度センサー
外部電流センサー
電圧センサー
```

推奨：

```text
ロードセル
温度センサー
```

要求：

```text
外部角度はsysidの正観測値として扱う
サーボ内部角度は補助観測値として扱う
```

---

## 4.6 電源条件

サーボの挙動は電源電圧で変化する。

要件：

```text
測定時の電圧を固定する
電圧変動をログに残す
電源電流制限値をログに残す
```

---

## 4.7 温度条件

サーボは温度で挙動が変わる。

マイルストーン1では、温度モデルを完全化しなくてもよい。

ただし、以下は必要。

```text
測定開始温度
測定終了温度
試験間の冷却時間
連続試験時間
```

---

## 4.8 安全停止条件

負荷試験では破損リスクがある。

必要：

```text
最大電流超過で停止
温度超過で停止
角度範囲超過で停止
通信断で停止
非常停止
```

---

## 4.9 MuJoCo反映方式

マイルストーン1では、まず外部トルクモデル方式がよい。

```text
指令角
↓
ActuatorBehaviorModel
↓
出力トルク
↓
MuJoCo jointへ反映
```

MJCFのposition actuatorだけに閉じると、デッドバンド・パンチ・保護動作を扱いにくい。

---

## 4.10 sysid結果の公開フォーマット

推奨パラメータを公開するなら、形式を決める必要がある。

例：

```text
device_profile.json
behavior_model.json
sysid_result.json
validation_report.md
```

---

## 4.11 URDF_Kitchen連携要件

URDF_Kitchen出力時にテンプレートからパラメータセットを選べるようにするには、最低限以下が必要。

```text
actuator_profile_id
device_category
vendor
model
parameter_set_id
mujoco_mapping
```

URDF_Kitchen側作業であっても、runtime側は参照可能な形式でパラメータセットを公開する必要がある。

# 10. 結論

今回提示された小目標・中目標・大目標は、これまで検討してきた構成に自然に当てはまる。

ただし、すぐに大目標へ進むのではなく、

```text
KRS-2552単体
↓
KHR-3HV全体
↓
歩行モーション双方向検証
↓
設定値テンプレート化
↓
sysid自動化
↓
他デバイス展開
↓
フィールド・タスク・HIL連携
```

の順で進めるのがよい。

特にMilestone 1では、

```text

```

ことに集中するべきである。
